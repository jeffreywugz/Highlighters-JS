Notes on the various categories:

Version/last updated:
Features change with new releases. The information in this comparison is only good for the versions specified. However, some projects are under more active development, so a more feature complete version may be available.
Framework:
While building on a JavaScript framework is not bad, it does limit the usefulness of a project. Some users may not want the extra bandwidth overhead required by a full framework. Other's may already be using a different framework and would rather not need to support another one.
Other Dependencies:
Some of the projects highlighted here build on top of others.
Serve Lang Defs Seperately:
Some projects are configured so that only the language definitions you actually need are downloaded, which saves bandwidth. This also allows you to provide your own language definitions and serve them on your site without contributing them upstream for inclusion in the base package. While you certainly are encouraged to contribute upstream, this allows immediately deployment without waiting for the next release. It also allows support of unusual one-off implementations without involving third parties or unsupported hacks.
Bundled styles:
While anyone can create their own CSS styles, it can be nice to get multiple styles to chose from out of the box. In the end though, probably not a very important factor. Although, some projects may serve multiple styles in one CSS file which wastes bandwidth if the unused styles are not deleted before deployment.
Code Block Syntax:
All projects work by scanning the html page for a specifically marked section of HTML which contains a code block and running the enclosed code block through its parser. Some projects only allow the code blocks to be marked up in one very rigid way. Others allow multiple variations of the HTML which constitutes a code block. Some even allow you to define your own code block syntax when initializing the program in JavaScript.
Auto-Detect language:
Some projects do not require a language to be specified for each code block and will try to guess. Of course, 100% accuracy can never be guaranteed, especially for shorter blocks.
Line Numbering:
Some projects offer a built-in feature to generate line numbers for code blocks. However, in some projects, you can only specify this feature in JavaScript - which in the common case means all code blocks on the page will get the same behavior. However, other projects may allow a per block behavior based on a clue in the block itself (usually a html class set on the containing block).
Copy plain text:
Your users will not want to manually remove line numbers or other styling cues from copied code blocks. However, there are a number of different ways this can be implemented:
To Clipboard:
Clicking a link provided in a caption/header will immediately copy the raw text to the OS's clipboard.
With Pop-Up:
Clicking a link in a caption/header will open a pop-up in the browser which contains the raw text to be selected and copied.
In-Line:
Clicking a link in a caption/header will toggle the styled text to raw text which can then be selected and copied.
By Default:
The styling is implemented in such a way that selecting and copying the styled text will always result in perfectly formatted raw text. In other words, no "copy plain text" feature is necessary. This generally (but not always) coincides with not having line numbers.
Redundant:
Some projects offer multiple of the above options. However, rather than using only one of the above at a time, they offer multiple choices to the site visitor as once, which is unnecessarily redundant and may confuse the user.
